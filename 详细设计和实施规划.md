# 1Remote UI 重构详细设计和实施规划

## 1. 目标

基于当前 fork 的 1Remote 项目进行二次开发，保留现有远程连接与数据管理能力，对 UI 进行系统性重构，使产品整体交互和视觉体验接近 VSCode 一类桌面工作台软件。

本次改造的核心原则如下：

- 保留现有协议能力、数据源能力、会话管理能力。
- 保留现有 WPF + Stylet + Dragablz 技术路线，不进行跨技术栈重写。
- 优先重构 Shell、布局、视觉体系和交互模型，避免先动底层连接逻辑。
- 分阶段实施，保证每个阶段都可编译、可运行、可回退。

## 2. 现状分析

### 2.1 技术栈现状

当前项目主体为 Windows 桌面应用：

- UI 技术：WPF
- 应用框架：Stylet / StyletIoC
- Tab 与拖拽能力：Dragablz
- 部分协议宿主：WindowsFormsHost / 原生控件承载
- 主题机制：`ThemeService` + `ResourceDictionary`

### 2.2 关键代码位置

- 应用入口与 IoC：`Ui/Bootstrapper.cs`
- 主应用资源：`Ui/App.xaml`
- 主窗口：`Ui/View/MainWindowView.xaml`
- 主窗口状态管理：`Ui/View/MainWindowViewModel.cs`
- 列表视图：`Ui/View/ServerView/List/ServerListPageView.xaml`
- 树视图：`Ui/View/ServerView/Tree/ServerTreeView.xaml`
- 设置页：`Ui/View/Settings/SettingsPageView.xaml`
- 编辑页：`Ui/View/Editor/ServerEditorPageView.xaml`
- 连接 Tab 窗口：`Ui/View/Host/TabWindowView.xaml`
- 主题服务：`Ui/Service/ThemeService.cs`
- 会话控制：`Ui/Service/SessionControlService*.cs`

### 2.3 当前架构优劣

优势：

- 业务能力和 UI 已有一定分层，协议连接逻辑并未全部写死在 XAML 中。
- 已具备主题切换、列表树切换、设置页、编辑页、标签页等基础能力。
- 已有 Dragablz，可复用其标签页、拖拽、分离窗口能力。

限制：

- 主窗口布局仍偏传统业务桌面应用，不是 IDE/工作台式 Shell。
- 多个页面存在较重的页面级布局代码，视觉系统不够统一。
- 主窗口、设置页、编辑页、会话窗口的视觉语言尚未形成统一设计系统。
- 会话承载涉及全屏、分辨率、多窗口逻辑，不能粗暴替换容器。

## 3. 重构目标界面

目标界面参考 VSCode 的工作台结构，但保留 1Remote 作为远程会话管理工具的业务特性。

### 3.1 目标布局

建议重构为五大区域：

1. Activity Bar
2. Side Bar
3. Editor Area
4. Bottom Panel
5. Status Bar

对应说明如下：

- Activity Bar：位于最左侧，提供“资源管理器 / 搜索 / 收藏标签 / 快速连接 / 设置 / 关于”等一级入口。
- Side Bar：根据 Activity Bar 当前选择切换内容，如服务器树、列表、标签筛选、数据源配置入口等。
- Editor Area：主工作区，用于显示远程连接标签页、服务器编辑器、设置页、说明页等。
- Bottom Panel：显示日志、传输、任务、连接诊断、脚本执行输出等辅助信息。
- Status Bar：显示当前数据源、筛选状态、连接状态、主题、版本、同步状态等全局状态。

### 3.2 目标交互

- 支持单窗口工作台模式。
- 支持多标签远程连接会话。
- 支持从资源区双击打开连接到编辑区。
- 支持命令面板，统一承载快速连接、搜索命令、切换主题、打开设置等动作。
- 支持资源区树形与列表模式自由切换。
- 支持底部面板展开/收起。
- 支持将连接会话分离为独立窗口，保留现有多窗口能力。

## 4. 总体设计

### 4.1 设计原则

- Shell 与业务页面解耦。
- 主题 Token 化，减少页面内直接写死颜色和间距。
- 页面容器标准化，统一标题栏、工具栏、侧栏、底栏、编辑区规范。
- 尽量复用现有 ViewModel，优先替换 View 和容器层。

### 4.2 新的 UI 分层建议

建议在现有 `Ui/View/` 基础上新增工作台层：

- `Ui/View/Workbench/`
- `Ui/View/Workbench/Controls/`
- `Ui/View/Workbench/Panels/`
- `Ui/View/Workbench/ActivityBar/`
- `Ui/View/Workbench/StatusBar/`

建议新增以下核心对象：

- `WorkbenchView`
- `WorkbenchViewModel`
- `ActivityBarItemViewModel`
- `SideBarViewModel`
- `EditorHostViewModel`
- `BottomPanelViewModel`
- `StatusBarViewModel`
- `CommandPaletteViewModel`

其中：

- `WorkbenchViewModel` 负责整体布局状态切换。
- `MainWindowViewModel` 可逐步退化为窗口级壳层或转为代理到 `WorkbenchViewModel`。
- 现有 `ServerListPageViewModel`、`ServerTreeViewModel`、`SettingsPageViewModel`、`ServerEditorPageViewModel` 作为工作台内容页挂载。

### 4.3 页面映射关系

现有页面到目标工作台区域的建议映射如下：

- `ServerListPageView` / `ServerTreeView` -> Side Bar 主资源区域
- `ServerEditorPageView` -> Editor Area 中的编辑器标签页
- `SettingsPageView` -> Editor Area 中的设置标签页
- `AboutPageView` -> Editor Area 中的信息页
- `TabWindowView` -> Editor Area 中的会话标签宿主，必要时可分离为独立窗口
- `LauncherWindowViewModel` / `QuickConnectionViewModel` -> 命令面板或快速打开入口

## 5. 详细模块设计

### 5.1 Workbench Shell

主工作台负责承载应用所有一级结构。

布局建议：

- 顶部保留自定义标题栏，但简化为 VSCode 风格窗口栏。
- 左侧固定 Activity Bar。
- 右侧采用三行结构：顶部 Editor Header，中部内容区，底部 Status Bar。
- 内容区再细分为 Side Bar、Editor Area、可选 Bottom Panel。

Shell 需要管理以下状态：

- 当前激活的 Activity
- Side Bar 是否显示
- Bottom Panel 是否显示
- 当前 Editor Tab 集合
- 当前激活的 Editor
- 当前主题
- 当前全局筛选关键字

### 5.2 Activity Bar

建议提供以下一级入口：

- Explorer：服务器资源
- Search：快速过滤和搜索
- Tags：标签与收藏
- Connections：当前连接会话
- Settings：设置
- About：关于

设计要求：

- 图标优先使用统一矢量资源。
- 支持选中态、悬浮态、提示文案。
- 支持未来扩展更多业务入口。

### 5.3 Side Bar

Explorer 模式下展示：

- 数据源选择器
- 树视图 / 列表视图切换
- 搜索过滤框
- 标签过滤区
- 服务器列表区

Search 模式下展示：

- 全局搜索输入框
- 匹配结果列表
- 常用筛选项

Tags 模式下展示：

- 已固定标签
- 常用标签
- 标签批量连接入口

Settings 模式下展示：

- 设置导航菜单
- 最近打开设置项

### 5.4 Editor Area

Editor Area 是新的主工作区，承载：

- 远程连接会话标签页
- 服务器编辑页
- 设置页
- 关于页
- 日志详情页

设计要点：

- 使用统一标签页头样式。
- 支持关闭、切换、固定、右键菜单。
- 会话标签页与表单编辑标签页视觉风格一致，但内容容器独立。
- 支持后续扩展双栏编辑、面板分组。

### 5.5 远程会话容器

该模块是风险最高的部分，建议谨慎重构。

原则如下：

- 第一阶段不重写 `SessionControlService`。
- 保持 `TabWindowView` 及其承载逻辑可用。
- 优先调整 Tab Header 外观与外围容器样式。
- 保留独立窗口、全屏、多显示器逻辑。

后续可考虑两种模式：

- 模式 A：主工作台内嵌会话标签页，独立窗口为辅。
- 模式 B：继续沿用独立 `TabWindowView` 作为会话工作区，主窗口主要负责资源管理。

建议优先采用模式 A，但实现时保留模式 B 作为回退方案。

### 5.6 Bottom Panel

建议新增底部面板，用于承载：

- 连接日志
- 文件传输状态
- 脚本执行输出
- 数据源诊断
- 错误与警告

设计要求：

- 可收起/展开。
- 支持多标签切换。
- 不影响主工作区焦点操作。

### 5.7 Status Bar

建议显示以下信息：

- 当前数据源名称
- 当前筛选模式
- 当前会话数量
- 当前主题
- 当前语言
- 软件版本
- 同步或数据库连接状态

### 5.8 命令面板

建议新增统一命令面板，承担以下能力：

- 快速搜索服务器
- 快速打开最近连接
- 执行“切换主题”“打开设置”“切换数据源”等命令
- 作为现有 `Launcher` 能力的升级入口

建议快捷键：

- `Ctrl+Shift+P` 打开命令面板
- `Ctrl+P` 快速搜索服务器
- `Ctrl+B` 显示或隐藏 Side Bar
- `Ctrl+J` 显示或隐藏 Bottom Panel

## 6. 主题与视觉系统设计

### 6.1 主题系统方向

当前 `ThemeService` 已支持颜色和字体动态资源，建议继续扩展，不替换机制。

建议将主题资源细化为工作台 Token：

- `WorkbenchBackgroundBrush`
- `ActivityBarBackgroundBrush`
- `ActivityBarForegroundBrush`
- `SideBarBackgroundBrush`
- `EditorBackgroundBrush`
- `PanelBackgroundBrush`
- `StatusBarBackgroundBrush`
- `TabActiveBackgroundBrush`
- `TabInactiveBackgroundBrush`
- `BorderSubtleBrush`
- `FocusAccentBrush`

### 6.2 视觉风格建议

- 默认提供 `VSCode Dark` 风格主题。
- 保留现有 Light 风格和品牌主题作为兼容主题。
- 统一控件圆角、边距、分割线、悬浮态、选中态。
- 尽量减少高饱和度大面积背景，用层次分明的灰阶结构提升专业感。

## 7. 实施规划

### 阶段 0：重构准备

目标：

- 梳理现有 UI 入口和容器关系。
- 明确哪些 ViewModel 可直接复用。
- 标记高风险模块。

输出：

- 页面映射表
- 风险清单
- 组件命名规范

### 阶段 1：建立 Workbench 壳层

目标：

- 新增 Workbench 容器。
- 将现有主窗口改造成工作台式布局。
- 保持原有核心功能仍可进入。

实施项：

- 新建 `WorkbenchView` 与 `WorkbenchViewModel`
- 调整 `MainWindowView`，只负责窗口宿主和标题栏
- 引入 Activity Bar、Side Bar、Editor Area、Status Bar 基础布局
- 先以占位内容接入现有页面

验收标准：

- 应用可启动
- 主窗口结构切换为新工作台布局
- 现有列表页和树页可被新壳层承载

### 阶段 2：资源管理区迁移

目标：

- 将服务器列表、树、标签筛选统一迁移到 Side Bar

实施项：

- 重构 `ServerListPageView`
- 重构 `ServerTreeView`
- 抽离通用搜索框、工具栏、筛选区
- 增加资源区视图切换按钮

验收标准：

- 用户可以在 Side Bar 中完成主要资源浏览和筛选
- 树形、列表、卡片模式仍可用

### 阶段 3：编辑区改造

目标：

- 将设置页、编辑页、说明页统一放入 Editor Area

实施项：

- 统一 Editor Tab 模型
- 将 `ServerEditorPageView` 接入编辑器标签页体系
- 将 `SettingsPageView` 改造成编辑器式页面
- 将 `AboutPageView` 改造成普通信息页

验收标准：

- 编辑页、设置页可作为标签页打开和关闭
- 打开逻辑一致，用户心智统一

### 阶段 4：连接会话工作区改造

目标：

- 将远程连接会话更自然地融入工作台

实施项：

- 优化 `TabWindowView` 的标签样式与容器边界
- 统一连接 Tab 与普通编辑 Tab 的交互规则
- 验证全屏、独立窗口、多显示器行为未回归

验收标准：

- 至少 RDP、SSH、VNC 三类典型连接行为正常
- 多标签切换、关闭、重连行为正常
- 全屏与恢复逻辑不退化

### 阶段 5：底部面板与命令面板

目标：

- 补齐 VSCode 风格工作台的辅助效率能力

实施项：

- 新增 Bottom Panel
- 新增命令面板
- 接入日志、任务和快速命令入口

验收标准：

- 快捷键可打开命令面板
- 面板可切换、收起、恢复

### 阶段 6：视觉统一与体验打磨

目标：

- 完成视觉体系收敛
- 提升一致性和专业感

实施项：

- 统一 spacing、font、icon、hover、active、border 规范
- 修复深色主题下对比度问题
- 优化键盘导航与焦点状态
- 清理冗余样式和局部写死颜色

验收标准：

- 视觉一致性明显提升
- 不同页面之间无明显风格割裂

## 8. 风险与应对

### 8.1 会话承载风险

风险：

- RDP/VNC 等宿主控件对窗口尺寸、焦点、全屏、宿主层级敏感。

应对：

- 先做外围容器改造，不先改连接核心。
- 每阶段都回归 RDP、VNC、SSH 连接测试。

### 8.2 XAML 复杂度风险

风险：

- 现有多个页面 XAML 较长，局部重构容易引入样式回归。

应对：

- 优先抽通用控件和样式字典。
- 每次只替换一个层级，不大面积同时改动。

### 8.3 交互迁移风险

风险：

- 用户已有使用习惯，若交互变化过大，学习成本会上升。

应对：

- 保留原有快捷连接路径。
- 提供兼容入口和渐进式布局迁移。

## 9. 测试与验收建议

每个阶段至少验证以下场景：

- 应用启动与退出
- 数据源切换
- 树形浏览、列表浏览、卡片浏览
- 搜索过滤
- 打开服务器编辑器并保存
- SSH 连接
- RDP 连接
- VNC 连接
- 多标签切换与关闭
- 独立窗口与全屏切换
- 设置页打开与主题切换

建议建立最小人工回归清单，后续再逐步补充自动化测试。

## 10. 结论

以当前 1Remote 项目为基础实施 VSCode 风格 UI 重构，技术上可行，且优于重写。

最优策略不是替换核心技术栈，而是在保留 WPF、Stylet、Dragablz 和现有会话控制逻辑的前提下，新增 Workbench Shell，逐步迁移资源区、编辑区、会话区与主题系统。

建议优先完成阶段 1 到阶段 3，这三步完成后，产品形态将已明显从传统远程管理工具演进为工作台式桌面应用，后续再继续攻克会话容器和高级交互。
